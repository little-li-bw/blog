# React + Spring Boot 个人博客系统设计文档

## 1. 文档状态

- 日期：`2026-04-18`
- 状态：`已批准，允许进入实现规划`
- 范围：用于简历展示的第一版个人博客网站
- 前端基线：现有 `frontend/` 下的 React 项目
- 后端目标：后续补充 `Spring Boot` 服务

## 2. 产品目标

构建一个可以部署到腾讯云，并能够挂在简历上的个人技术博客。

它不是通用博客平台，也不是社区产品。核心目标是：

- 以 Java 后端候选人的身份展示站点所有者
- 通过文章体现技术深度
- 保持首页信息密度高、路径短、面试官易于快速扫描
- 保持后端规模可由单人独立完成和维护

第一版必须足够完整，可对外使用，但功能边界要刻意收紧。

## 4. 信息架构

公网只保留两个页面：

1. 首页
2. 文章详情页

这是有意为之。产品面向简历流量优化，首次访问必须直接展示身份、技术栈和写作输出，不能让访客在多个页面之间来回跳转。

## 5. 当前前端基线

现有前端已经给出了明确的视觉方向。后续实现必须延续这个方向，而不是被通用后台模板风格替换。

### 5.1 需要保留的视觉语言

- 黑白灰与石板色为主的配色
- 放大的首屏标题
- 粗黑标题与标签
- 高对比卡片布局
- 少量强调色
- 固定顶部导航和滚动状态变化
- 宽松的纵向节奏
- 以入场动画为主的轻运动
- 更像编辑式作品集，而不是管理后台

### 5.2 当前定义风格的 React 文件

- `frontend/src/pages/Home.tsx`
- `frontend/src/pages/BlogDetail.tsx`
- `frontend/src/components/Layout.tsx`
- `frontend/src/components/BlogCard.tsx`
- `frontend/src/index.css`

### 5.3 当前前端与最终范围的偏差

当前 React 项目仍然保留了独立 `/blog` 路由和项目相关示例数据，这与已确认产品范围不一致。

因此实现时必须：

- 把博客列表行为并回首页
- 移除对独立 `/blog` 页的长期依赖
- 移除公网项目展示内容
- 在保留视觉识别的同时，把路由收敛成：
  - `/`
  - `/posts/:id`

## 8. 技术架构

### 8.1 仓库结构

推荐结构：


### 8.2 前端技术栈

- React
- Vite
- TypeScript
- Tailwind CSS
- `motion`
- `react-router-dom`
- `react-markdown`

前端应保持内容站点属性，动画轻量，避免工具型后台风格。

#### 前端架构约束

- 路由数量保持最小
- 布局、页面、API、内容渲染职责清晰分离
- 用后端 API 数据替换本地 Mock 常量
- 没有功能必要时，不改变现有视觉 token 和 spacing 体系
- 不引入与当前风格冲突的通用 UI 库

### 8.3 后端技术栈

- Java
- Spring Boot
- Spring Web
- Spring Security
- JWT
- MyBatis-Plus
- MySQL
- Maven

后端采用单体部署，不拆微服务。

#### 后端模块边界

- `auth`：管理员登录与令牌签发
- `post`：文章公网与后台操作
- `category`：分类管理
- `tag`：标签管理
- `site`：站点配置
- `upload`：图片上传
- `stats`：阅读量

### 8.4 部署栈

- 腾讯云 Linux 服务器
- Docker Compose
- Nginx
- HTTPS 证书

建议路径：

- `/` -> React 前端
- `/api` -> Spring Boot 后端

## 10. API 设计

### 10.1 公网 API

#### `GET /api/posts`

用途：

- 首页文章列表

查询参数：

- `keyword`
- `categoryId`
- `tagId`
- `pageNum`
- `pageSize`

行为：

- 只返回已发布文章
- 第一版搜索只按标题匹配

#### `GET /api/posts/{id}`

用途：

- 获取文章详情

行为：

- 自增阅读量
- 返回上一篇和下一篇文章元信息

#### `GET /api/categories`

用途：

- 首页分类筛选数据源

#### `GET /api/tags`

用途：

- 首页标签筛选数据源

#### `GET /api/site-config`

用途：

- 首页个人信息与站点文案

### 10.2 后台 API

#### `POST /api/admin/login`

- 登录并返回 JWT

#### `GET /api/admin/posts`

- 后台文章列表

#### `POST /api/admin/posts`

- 创建文章

#### `PUT /api/admin/posts/{id}`

- 更新文章

#### `DELETE /api/admin/posts/{id}`

- 删除文章

#### `PUT /api/admin/posts/{id}/status`

- 切换草稿、发布、下线状态

#### `GET/POST/PUT/DELETE /api/admin/categories`

- 分类管理

#### `GET/POST/PUT/DELETE /api/admin/tags`

- 标签管理

#### `GET /api/admin/site-config`

- 获取可编辑站点配置

#### `PUT /api/admin/site-config`

- 更新站点配置

#### `POST /api/admin/upload`

- 上传正文内联图片

## 12. 非功能要求

### 12.1 性能

- 首页首屏必须保持轻量
- 文章列表 API 必须分页
- 渲染 HTML 在浏览器侧可利用缓存
- 上传图片必须限制体积

### 12.2 安全

- 后台 API 必须使用 JWT
- 密码必须安全哈希
- 上传文件必须校验类型和大小
- 公网 Markdown 渲染必须做安全过滤

### 12.3 可靠性

- 后端统一返回格式
- 全局异常处理必须存在
- 非法文章 ID 返回清晰 404
- 前端必须提供列表和详情失败时的优雅降级状态

### 12.4 SEO

- 首页标题和描述
- 文章级标题和摘要元信息
- 语义化标题结构
- 可读性良好的文章 URL

## 14. 风险与决策

### 已确认决策

- 前端继续使用 React
- 视觉方向沿用当前前端
- 公网只保留首页和文章详情页
- 不做项目页
- 不做评论
- 不做精选文章
- 不做封面图

### 主要风险

- 当前前端还保留了无用路由和示例结构，如果不尽早清理，会污染后续实现
- 如果首页不断堆内容而不做分页和区块边界，信息密度会变成信息拥挤
- 当前源码中的占位数据和乱码文本必须在接入真实内容前清理

## 15. 验收标准

第一版完成的标准是：

- 访客打开首页后，能快速理解站点所有者是谁
- 首页可搜索、可筛选文章
- 首页文章列表直接暴露分类和标签
- 访客可以在桌面端和移动端清晰阅读文章详情
- 管理员可以登录并管理文章、分类、标签和首页配置
- 网站可以稳定部署到腾讯云并提供公网访问地址

## 16. 实现注意事项

- 不要重新引入项目页，也不要通过导航变相恢复它
- 不要再增加独立博客列表页，除非后续产品边界明确变化
- 当前 React 前端就是视觉契约
- 把现有前端视作设计基线，不要直接当成最终业务逻辑
