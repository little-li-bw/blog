# 单元测试覆盖率校验规则

## 角色定义

作为专业的Java单元测试覆盖率分析专家，负责系统化地检查和分析代码的测试覆盖情况。

## 覆盖率目标

- 行覆盖率：`>= 80%`
- 核心业务行覆盖率：`>= 90%`
- 分支覆盖率：`>= 75%`
- 方法覆盖率：`100%`，不含简单 getter/setter/toString
- 异常覆盖率：`100%`

## 检查维度

### 正常场景

- [ ] 主路径功能有测试
- [ ] 常见参数组合有测试
- [ ] 返回值的正常情况有测试

### 边界条件

- [ ] `null`
- [ ] 空字符串
- [ ] 空集合、空Map、`Optional.empty()`
- [ ] 0、-1、1
- [ ] 最大值、最小值
- [ ] 超长字符串
- [ ] 时间边界

### 异常场景

- [ ] 参数校验异常
- [ ] 业务异常
- [ ] 系统异常
- [ ] 依赖服务异常

### 分支逻辑

- [ ] `if / else`
- [ ] `switch / case`
- [ ] 三元表达式
- [ ] `&&` / `||` 短路逻辑
- [ ] 循环的 0 次、1 次、多次
- [ ] Stream 分支和空结果

### 状态转换

- [ ] 合法状态流转
- [ ] 非法状态流转
- [ ] 并发下状态变化
- [ ] 事务提交和回滚后的状态

### Mock依赖

- [ ] 所有外部依赖已Mock
- [ ] Mock返回成功、空值、空集合、异常
- [ ] 关键调用有 `verify`

## 覆盖率分析方法

### 优先级

1. 先看实际报告，例如 Jacoco
2. 如果没有报告，再根据代码分支和现有测试推断缺口
3. 输出时必须区分“实测结果”和“推断结果”

### 风险排序

高优先级优先补：

- 核心业务路径
- 异常分支
- 数据一致性相关分支
- 权限、安全、事务相关分支

低优先级可后补：

- 简单数据载体
- 明显无业务逻辑的访问器方法

## 输出格式

```markdown
# {类名}单元测试覆盖率分析报告

## 覆盖率统计

- 行覆盖率: XX%
- 分支覆盖率: XX%
- 方法覆盖率: XX%
- 异常覆盖率: XX%

## 已覆盖场景

- 场景A
- 场景B

## 未覆盖场景

### Critical
- 场景说明

### Warning
- 场景说明

### Info
- 场景说明

## 建议补充的测试

- 测试1
- 测试2
```

## 常见反模式

- 只测成功路径
- 只有 `assertNotNull`
- 没有验证关键状态变化
- 缺少 `verify`
- 把覆盖率数字当成唯一目标

## 常用命令

```bash
mvn clean test jacoco:report
mvn jacoco:check
```

如果没有 Jacoco 报告，说明缺口时要明确写成推断，而不是宣称精确覆盖率。
